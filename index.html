A Financial and Project Management-Based Execution-Risk Framework for Mumbai Real Estate Projects

The framework models the complete 6-stage development lifecycle with zone-aware costing, FSI/DR/TDR strategy optimization, funding waterfall analysis, dynamic risk scoring, and side-by-side scenario comparison across Mumbai's nine micro-market zones.

Core Objective

Build a structured, repeatable decision-making framework for evaluating feasibility and execution risk of real estate development projects in Mumbai. The model serves three audiences: developers assessing project viability, lenders evaluating financing exposure, and academic committees evaluating methodological rigor.

04 · Phase 1: Problem Structuring & Research

January 13 – February 1, 2026

Narrative & key decisions

The project began with the spring semester on January 13. The first two weeks were spent defining the problem space, conducting a literature review on real estate project risk frameworks, and engaging our industry partner. I reached out to Webtech Realty in Mumbai to secure access to structured project data from their completed and ongoing developments, including The Presidential (a major commercial project).

The pre-proposal was submitted on January 17, followed by the full written proposal on January 27. The oral proposal presentation established the scope and methodology for the capstone committee.

Key Decision: Stage-Gated Architecture

Chose a 6-stage gate model (Feasibility, Land Lock-in, Regulatory, CC Approval, Construction, Completion) over a continuous timeline approach. This aligned with how developers, lenders, and regulators in Mumbai actually structure project progression. Every subsequent model version preserved this architecture.

Architecture Decision: Isolated Compare Contexts

The financial engine carries extensively shared global state that is tightly coupled through the main recalculation path. Refactoring for multi-instance support would have required a major rewrite. Instead, each compare session runs in its own isolated context, giving complete state isolation without touching the core engine. Cross-context communication uses the browser’s standard message-passing API.

Cross-Context Score Sync

The initial implementation polled child DOMs directly, which worked when served over HTTP but browsers block cross-origin DOM access on the file:// protocol. Fixed by switching to postMessage: each session posts its updated score to the parent, which listens for the message and updates the gauge. Works universally regardless of protocol.

Also added per-session “Sample” test buttons in the compare header, and fixed navigation so clicking “Overview” while compare sessions are running re-opens the compare overlay instead of navigating away.

Audit narrative & context

With Compare Mode stable, attention shifted from feature expansion to engine correctness. Field-testing against Webtech Realty’s deal pipeline flagged a handful of areas where the engine’s output diverged from experienced-developer expectations on Mumbai RE conventions. Rather than patch each symptom, Phase 9 was run as a structured audit-then-fix pass over two weeks.

The phase opened (Apr 6–8) with a latent Compare Mode fix — replacing the V5.7.2 single-shot close flag with a sticky version that holds until explicit reopen — and the propagation audit itself: a 50-input perturbation matrix with recalculation-counter instrumentation that confirmed every canonical input flows through the engine. Apr 9–14 ran the code-read audit grouped by concern (stage cost and interest loop, stamp duty and regulatory costs, return KPIs, sensitivity tornado), each block checked against its statutory or market reference. Apr 15–18 cross-validated draft findings against live Webtech Realty deals on the V5.7.2 dashboard, reclassifying rounding-band artefacts as UX rather than engine items and stabilising the catalogue at ten findings (four HIGH, six MEDIUM). Apr 19 batched six refinements into V5.7.3 as an in-place edit to the same canonical filename. Apr 20 shipped V5.8 as a new file with five more refinements, per-stage soft-cost inputs, and the calendar-axis cash-flow rebuild. Apr 21 added a one-click analytics PDF export on the same V5.8 file — a read-only feature that captures the full analytics page (all five Chart.js figures, the sensitivity tornado, and the cash-flow table) into a branded multi-page PDF, session-aware in Compare Mode.

V5.8 Headline: Calendar-Axis Cash-Flow Rebuild

V5.8’s most significant structural change. The equity cash-flow generator was rebuilt on a calendar-month axis so concurrent stages (common in phased Redevelopment where S2↔S3, S3↔S4, and S5↔S6 overlaps compress the schedule) share the same calendar slot rather than accumulating serially. The earlier serial-axis treatment silently inflated project length whenever stages overlapped. On overlap-heavy projects the rebuild can shift the reported cash-flow IRR by 200–400 basis points — a meaningful difference for leveraged deal screening.

Two markdown reports live alongside this log for reproducibility: audit_v572_propagation.md (the Apr 8 black-box audit with the 50-input perturbation matrix) and MASTER_PROMPT_V572_SESSION3.md (the Apr 19 code-read audit with the ten-finding catalogue and approved edit plan). Each refinement in V5.7.3 and V5.8 traces back to a specific line-numbered entry in those reports.

The prior serial-axis treatment silently double-counted interest whenever stages overlapped — common on phased Redevelopment projects typical of Mumbai, where Stage 2↔3, Stage 3↔4, and Stage 5↔6 overlaps compress the schedule by three to six months.

AreaTreatment
Equity cash-flow axisCarried forward from V5.8 unchanged
Debt-accrual axisRebuilt to share the equity calendar axis rather than a stage-loop aggregate
Overlap interestConcurrent stages now share a calendar slot; no double-counting on overlap months
Bridge & delay interestPreserved per-stage (stage-local semantics retained)

Regression invariant confirmed. On zero-overlap projects V6.0 reproduces V5.8’s total interest to the paisa. On a representative 400 Cr / 500 Cr-debt Redevelopment project with a six-month Stage 3–4 overlap, V6.0 reports a cleaner total interest roughly ~2 Cr lower than V5.8 — the previously double-counted overlap window.

V6.1 — FSI Composition Auto / Override (Apr 24)

Industry feedback surfaced that real Mumbai plots often carry FSI breakdowns the standard DCPR 2034 road-width tables don’t capture — Cluster Redevelopment under §33(9), ODC under §33(20B), site-specific SRA incentives, and heavy TDR loading are common examples. V6.1 adds a new FSI Composition block inside the FSI Calculator card with a radio toggle between Auto and Override modes.

The pie chart and cost-breakdown table were extended with an Ancillary component. Downstream BUA, saleable area, and Development Rights area propagate reactively on any composition change. Auto-mode calibration is untouched, so existing saved projects continue to reproduce the same numbers.

Parallel UX Track (V6.x)

Two UX items noted during Phase 9 — integer-month rounding on duration/delay/overlap inputs, and debt-facility headroom absorbing small perturbations — remain on the V6.x backlog. Candidate affordances: integer snap on rounded inputs, a “drawn vs. facility” badge on the debt input, and scenario-gated dimming of inputs that are inert under the current project type. Sequenced after the V6.0 / V6.1 engine and FSI work to avoid reworking UI against a shifting baseline.

V6 Release Criteria — Met

V6 was declared shipped when (a) the end-to-end regression suite passed with zero KPI delta on zero-overlap projects, (b) overlap-heavy reference projects showed the expected cleaner total interest, and (c) allocation integrity held — the sum of per-stage interest matches the total-interest aggregate exactly on every scenario tested. All three criteria were met on Apr 24; V6.1 replaces V5.8 as the production file, and V5.8 is retained as the penultimate rollback target behind V5.7.2.

18 · Lessons Learned

Architecture: Simplicity Wins

The 6-stage gate structure was chosen on day one and never changed. The V5.0 experiment with 10 finer-grained phases confirmed a counterintuitive principle: more granularity does not always produce better insight. Building the entire dashboard as a single HTML file was unconventional but practical: it can be opened in any browser, shared as an email attachment, embedded in Electron, or served from a static host with zero dependency management.

Iframe Isolation Over Refactoring

When Compare Mode required running two independent sessions, the instinct was to refactor the dashboard into an instance-based architecture. Instead, we chose iframe isolation: each session runs in its own iframe with complete state separation. The cost was postMessage-based IPC for dark mode sync, score updates, and sample triggers. The benefit was zero risk of disturbing the financial engine. Sometimes the pragmatic architecture beats the elegant one.

Dual-Track Validation

Maintaining both Excel and HTML versions throughout provided a critical cross-validation mechanism. When the HTML dashboard produced a different result from the Excel model, it always pointed to a bug in one or the other.

The Cost of Not Testing

The V5.5 broken build was a turning point. The 574-check test suite for V5.6.1 paid for itself immediately when V5.6.2 code cleanup needed regression verification.

Protocol Matters

The score gauge bug in V5.7.2 taught a valuable lesson: contentDocument access works fine on http:// but is blocked on file:// due to browser security policies. The fix (postMessage) works universally. Always test on the actual deployment protocol, not just localhost.

Domain Expertise Shapes Software

The FSI/DR/TDR module illustrates how industry knowledge transforms a model. Conversations with Mumbai development professionals revealed the complexity of BMC premium financing, TDR market dynamics, and regulatory nuances that no amount of pure engineering could have anticipated.

Black-Box and White-Box Audits Catch Different Bugs

Phase 9 demonstrated that the two audit techniques are complementary, not redundant. The April 8 live-perturbation audit (black-box) could prove every input was wired and could flag absRate as propagation-suspect, but it could not see why the logic was wrong inside recalculate(). The April 19 code-read audit (white-box) caught the absorption-cap issue, the pre-sale debt double-service, and the stamp-duty base mis-specification — none of which would surface from input-output diffing alone. The rule that emerged: perturb first to narrow the suspect set, then read the code to diagnose. Running them sequentially found five fixable findings where either one alone would have found two.

Calendar Axes Matter for Financial Engines

The V5.8 real cash-flow IRR rewrite reinforced a lesson that applies to any multi-period financial model: the calendar time axis has to be a first-class concept, not a byproduct of iterator arithmetic. In a stage-gated project with overlapping phases — typical of Mumbai phased Redevelopment — a serial time axis and a calendar time axis can disagree by a month or two, which is enough to materially shift discounted metrics. Making the axis explicit fixed that class of subtle drift at the source rather than per-KPI.

Field Testing Reveals Silent Failures

Webtech Realty began using the V5.7.2 dashboard for live deal screening in early April. Within days, feedback surfaced the “I changed a parameter and nothing happened” complaint — which audit work eventually traced to integer-month rounding on stage-duration inputs and to the debt facility ceiling absorbing small perturbations. These were correct financial behaviors with insufficient UI affordance. A user who trusts the tool and sees silent behavior will stop trusting the tool. The V6 roadmap now includes affordances (scenario-gated input dimming, a “drawn vs facility” badge, integer snap) that pure code review would never have prioritized.
